## Spring JDBC (W2) vs Spring Data JPA (W3)

### BookRepository (W2)
- На второй неделе в JDBC я вручную прописывал SQL код для каждого метода репозитория и какие значения передаваемых параметров подставлять и в какую очередь, будь то перечислением подставляя под "?", будь то через мапу
- Занимал 1 метод минимум 5, максимум - 16 строк кода

### TaskRepository (W3)
- На третьей неделе репозиторий строю через JPA, а-ля  
`List<Task> findByStatus(TaskStatus status);`  
который использует ключевые слова `And`, `Before`, `count...By`, `exists...By` в названии метода, из-за чего достаточно согласно структуре составить название метода и JPA сам создаст SQL запрос 
- Так же имплементируя интерфейс `JpaRepository<Entity, UUID>` бесплатно получаю реализацию базовых методов а-ля `findById`, `findAll`, `save`, `delete`, `count`, `existsById`

### Вывод
- Магия заключается в ключевых словах и соблюдении структуры JPA, который в свою очередь понимает, исходя из структуры написанного названия метода, что конкретно мы хотим намутить в виде SQL запроса
- Но цена магии заключается в том, что мы не знаем что конкретно он делает под капотом, и за этим есть риск не уследить  
- Также есть проблема в технических ценах - Hibernate генерирует SQL сам, и нет гарантии что он делает это всегда самым эффективным способом, к тому же для методов со сложной логикой реализацию через название с помощью JPA по-человечески не реализовать, название вида `findByProjectIdAndStatusAndPriorityAndDueDateBefore` уже нечитаемо, поэтому надо писать кастомные запросы которые уже могут растягиваться
