## N + 1
### 1. Что такое N + 1 и почему так называется
Это явление при котором для получения данных из бд сначала делается первый общий запрос на получение данных, а после - дополнительные запросы на каждый объект для подгрузки "ленивых" данных.  
Поэтому так и называется: 1(первый запрос на получение общих данных) + N(дополнительные запросы на загрузку остальных данных при обращении к ним)

### 2+3. Четыре способа решения - JOIN FETCH, EntityGraph, BatchSize, DTO projection. Какой когда использовать?
1) JOIN FETCH - оператор при котором мы даем инструкцию Hibernate догрузить ленивые данные дочерних ассоциаций. Используется тогда, когда надо превратить ленивую загрузку в жадную. Гибкий, позволяет на каждый метод писать свой запрос, что одновременно является и минусом метода.
2) EntityGraph - по сути то же самое что и JOIN FETCH, но без JPQL. Удобный когда хочу просто переиспользовать стандартные методы а-ля findAll(), findById() и тд без написания запросов.
3) BatchSize - срабатывает в момент первого обращения к коллекции и подгружает следующие N id из тех, что уже загружены в persistence context. То есть когда в цикле впервые обращаемся к ленивой коллекции первого объекта - Hibernate не делает один запрос за коллекцию, а берет сразу N кол-во объектов из persistence context и загружает одним запросом. Является другим способом решения, если первые два метода думают как догрузить все сразу превращая ленивую загрузку в жадную - здесь мы не паримся и делаем загрузку сразу большого количества ленивых вещей. Из минусов - @BatchSize на @OneToOne не поддерживается начиная с Hibernate 6+
4) DTO projection - способ решения если возникает вопрос вида:"Зачем нам в принципе грузить и следить за всем, если нам нужно 2 поля на чтение?". Главная идея - загрузить только нужные поля для того чтобы их прочитать и отдать с инструкцией для Hibernate:"Вот эти поля не отслеживай, они 100% изменяться не будут". Удобный так как все реализуется через интерфейсы и стандартные Java record`ы.

### 4. Что такое OSIV и почему выключаем.
OSIV - паттерн при котором Hibernate держит открытым persistence context на протяжении всего HTTP-запроса, из-за чего мы можем не уследить что Hibernate делает кучу запросов в БД. Выключая - мы начинаем ловить LazyInitializationException, что заставляет нас правильно писать методы и экономить ресурс соединения HikariCP, так как этот ресурс держится открытым на весь HTTP-запрос включая сериализацию ответа, из-за чего при высокой нагрузке пул исчерпывается и новые запросы висят в ожидании.
